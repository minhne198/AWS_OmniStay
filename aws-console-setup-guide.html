Nguyên tắc quan trọng: làm tới đâu chụp màn hình tới đó. Trước khi tạo tài nguyên tốn phí, bật AWS Budget. Sau buổi lab, kiểm tra Billing và cleanup ALB, NAT Gateway, EC2, RDS, ElastiCache nếu không dùng nữa.

Bước 1

Chuẩn bị tài khoản, region và AWS Budgets

Bắt buộc
  1. Đăng nhập AWS Console bằng IAM user hoặc IAM Identity Center, không dùng root account nếu có thể.
  2. Chọn region cố định cho toàn bộ bài demo, gợi ý ap-southeast-1.
  3. Vào Billing and Cost Management → Budgets → Create budget.
  4. Chọn Cost budget, đặt ngưỡng demo ví dụ 5 USD hoặc 10 USD.
  5. Thêm email nhận cảnh báo ở mức 50%, 80%, 100%.

Kết quả cần có

AWS Budget đã active và có email cảnh báo chi phí.

Ảnh cần chụp

  • Trang Budget summary.
  • Màn hình region đang chọn.
  • Billing dashboard sau khi tạo budget.

Lưu ý chi phí

ALB, RDS, ElastiCache, NAT Gateway và public IPv4 có thể phát sinh phí theo giờ.

Bước 2

Tạo VPC, Subnets, Route Tables và Internet Gateway

Network
  1. Vào VPC → Your VPCs → Create VPC.
  2. Chọn VPC only, đặt tên omnistay-vpc, IPv4 CIDR 10.0.0.0/16.
  3. Tạo 2 public subnets cho ALB và EC2 demo:
    • 10.0.1.0/24 - public-a
    • 10.0.2.0/24 - public-b
  4. Tạo 2 private app subnets nếu muốn production-like sau này:
    • 10.0.10.0/24 - private-app-a
    • 10.0.11.0/24 - private-app-b
  5. Tạo 2 private DB subnets cho RDS và Redis:
    • 10.0.20.0/24 - private-db-a
    • 10.0.21.0/24 - private-db-b
  6. Vào Internet Gateways → tạo omnistay-igw → attach vào VPC.
  7. Tạo public route table, thêm route 0.0.0.0/0 trỏ tới Internet Gateway, associate với 2 public subnets.
  8. Không tạo NAT Gateway trong demo low-cost trừ khi mentor yêu cầu EC2 chạy private subnet.

Kết quả cần có

1 VPC, 6 subnet, public route table có route ra Internet Gateway.

Ảnh cần chụp

  • VPC details với CIDR 10.0.0.0/16.
  • Danh sách subnets theo 2 AZ.
  • Route table public có route tới IGW.

Quyết định demo

Để tiết kiệm, EC2 app có thể nằm public subnet nhưng chỉ nhận inbound từ ALB.

Bước 3

Tạo Security Groups theo từng layer

Security
  1. Vào EC2 → Security Groups → Create security group.
  2. Tạo SG-ALB: inbound HTTP port 80 từ 0.0.0.0/0. Nếu dùng HTTPS/domain sau này, thêm port 443.
  3. Tạo SG-EC2: inbound port 8080 chỉ từ source là SG-ALB.
  4. Tạo SG-RDS: inbound MySQL port 3306 chỉ từ source là SG-EC2.
  5. Tạo SG-Redis: inbound Redis port 6379 chỉ từ source là SG-EC2.
  6. Không mở SSH public nếu không cần. Nếu cần debug, dùng SSM Session Manager hoặc chỉ mở SSH tạm thời từ IP cá nhân rồi đóng lại.

Kết quả cần có

Traffic đi đúng chuỗi: Internet → ALB → EC2 → RDS/Redis.

Ảnh cần chụp

  • Inbound rules của SG-ALB.
  • Inbound rules của SG-EC2 với source là SG.
  • Inbound rules của SG-RDS và SG-Redis.

Không làm

Không mở RDS port 3306 hoặc Redis port 6379 từ 0.0.0.0/0.

Bước 4

Tạo Amazon RDS for MySQL

Database
  1. Vào RDS → Subnet groups → tạo subnet group omnistay-db-subnet-group gồm 2 private DB subnets.
  2. Vào RDS → Databases → Create database.
  3. Chọn Standard create, engine MySQL, version MySQL 8.0.
  4. Templates: chọn Free tier nếu tài khoản còn hỗ trợ, hoặc Dev/Test cấu hình nhỏ.
  5. DB identifier: omnistay-mysql, master username: admin.
  6. Đặt mật khẩu mạnh, lưu vào SSM Parameter Store hoặc Secrets Manager. Không lưu mật khẩu trong README, code hoặc ảnh chụp.
  7. Instance class: chọn loại nhỏ phù hợp ngân sách, ví dụ db.t3.micro nếu còn phù hợp region/tài khoản.
  8. VPC: chọn omnistay-vpc, subnet group: omnistay-db-subnet-group.
  9. Public access: No.
  10. Security group: chọn SG-RDS.
  11. Database name: hotel_booking.
  12. Sau khi status Available, lưu endpoint dạng host:3306.

Kết quả cần có

RDS MySQL available, private, chỉ EC2 app truy cập được qua SG-RDS.

Ảnh cần chụp

  • Database status Available.
  • Connectivity & security: endpoint, port, VPC, subnet group.
  • Security group gắn với RDS.
  • Backup/snapshot setting nếu bật.

Lưu ý

Single-AZ tiết kiệm cho demo. Multi-AZ dùng để thuyết minh HA nhưng có thể tăng chi phí.

Bước 5

Tạo Amazon ElastiCache Redis OSS hoặc Valkey

Cache
  1. Vào ElastiCache → tạo subnet group cho cache, chọn 2 private DB subnets.
  2. Vào ElastiCache → Create cluster.
  3. Chọn Redis OSS hoặc Valkey theo ngân sách. Plan cho phép Redis OSS hoặc Valkey.
  4. Cluster mode: disabled cho demo đơn giản.
  5. Node type: chọn node nhỏ phù hợp ngân sách, ví dụ cache.t3.micro nếu khả dụng.
  6. Replicas: 0 cho demo tiết kiệm; thêm replica nếu cần thuyết minh HA.
  7. VPC: omnistay-vpc, subnet group: cache subnet group vừa tạo.
  8. Security group: chọn SG-Redis.
  9. Sau khi status Available, lưu primary endpoint và port 6379.

Kết quả cần có

Redis/Valkey cluster available trong private subnet, backend EC2 có thể kết nối.

Ảnh cần chụp

  • Cluster status available.
  • Primary endpoint.
  • Subnet group và security group.
  • Metric cache trong CloudWatch nếu có request.

Demo cache

Search lần đầu là cache MISS, search lại cùng điều kiện là cache HIT. Đây là bằng chứng dùng ElastiCache.

Bước 6

Tạo IAM Role và lưu cấu hình bằng SSM hoặc Secrets Manager

Identity
  1. Vào IAM → Roles → Create role.
  2. Trusted entity: AWS service, use case: EC2.
  3. Role name: omnistay-ec2-role.
  4. Gắn quyền tối thiểu:
    • Đọc object trong S3 artifact bucket.
    • Đọc SSM Parameter Store hoặc Secrets Manager.
    • Ghi CloudWatch Logs.
    • SSM Managed Instance Core nếu muốn dùng Session Manager.
  5. Vào Systems Manager → Parameter Store hoặc Secrets Manager.
  6. Tạo parameter/secret cho DB password, RDS connection string, Redis endpoint, app environment.
  7. Đặt dạng SecureString nếu dùng Parameter Store cho giá trị nhạy cảm.

Kết quả cần có

EC2 đọc được cấu hình runtime mà không cần hard-code password hoặc access key trong code.

Ảnh cần chụp

  • IAM role attached policies.
  • EC2 trust relationship.
  • Danh sách parameters/secrets, che value nhạy cảm.

Không làm

Không đưa AWS access key, DB password, Redis endpoint private vào GitHub công khai.

Bước 7

Tạo Amazon S3 bucket cho frontend và artifact

Storage
  1. Vào S3 → Create bucket.
  2. Tạo frontend bucket, ví dụ omnistay-frontend-2026-<random>.
  3. Region: cùng region đã chọn.
  4. Object ownership: ACLs disabled.
  5. Block Public Access: bật toàn bộ.
  6. Không bật Static Website Hosting cho bucket này vì CloudFront OAC cần S3 REST origin.
  7. Upload thư mục frontend đã build/tĩnh vào bucket, gồm public/index.html, public/config.js, src/*.js, src/*.css.
  8. Nếu dùng artifact riêng cho backend, tạo bucket hoặc prefix riêng, ví dụ artifacts/api/app.zip.

Kết quả cần có

S3 bucket private, không public, CloudFront sẽ là đường truy cập chính.

Ảnh cần chụp

  • Bucket Block Public Access bật.
  • Danh sách objects frontend.
  • Bucket policy sau khi CloudFront OAC được gắn.

Config frontend

Khi chạy qua CloudFront, config.js nên đặt API_BASE: "/api".

Bước 8

Tạo CloudFront Distribution với Origin Access Control

CDN
  1. Vào CloudFront → Create distribution.
  2. Origin domain: chọn S3 bucket frontend dạng REST endpoint, không dùng website endpoint.
  3. Origin access: chọn Origin access control settings → tạo OAC mới.
  4. Viewer protocol policy: Redirect HTTP to HTTPS nếu muốn demo HTTPS ở CloudFront.
  5. Default root object: public/index.html.
  6. Sau khi tạo distribution, CloudFront sẽ đề xuất bucket policy. Copy policy đó vào S3 bucket policy.
  7. Thêm origin thứ hai là ALB DNS cho backend API.
  8. Thêm behavior /api/*:
    • Origin: ALB.
    • Allowed methods: GET, HEAD, OPTIONS, PUT, POST, PATCH, DELETE.
    • Cache policy: caching disabled hoặc TTL thấp cho API.
    • Origin request policy: forward headers/query string cần thiết.

Kết quả cần có

CloudFront phục vụ frontend từ S3 private và forward /api/* về ALB.

Ảnh cần chụp

  • Distribution status Deployed.
  • Origins: S3 + ALB.
  • Behavior mặc định và behavior /api/*.
  • OAC settings.
  • S3 bucket policy cho CloudFront.

Không làm

Không public S3 bucket và không dùng S3 Static Website Hosting nếu mục tiêu là OAC.

Bước 9

Tạo ALB, EC2 Launch Template và Auto Scaling Group

Compute
  1. Vào EC2 → Target Groups → tạo target group:
    • Target type: Instances.
    • Protocol: HTTP.
    • Port: 8080.
    • Health check path: /health.
  2. Vào Load Balancers → tạo Application Load Balancer:
    • Scheme: Internet-facing.
    • Subnets: 2 public subnets.
    • Security group: SG-ALB.
    • Listener HTTP 80 forward tới target group.
  3. Vào Launch Templates → tạo template cho EC2 app:
    • AMI: Amazon Linux 2023 hoặc AMI phù hợp .NET runtime.
    • Instance type: nhỏ phù hợp ngân sách, ví dụ t3.micro nếu khả dụng.
    • IAM instance profile: omnistay-ec2-role.
    • Security group: SG-EC2.
    • User Data: cài runtime, tải artifact từ S3, đọc SSM/Secrets, start service API.
  4. Vào Auto Scaling Groups → tạo ASG:
    • Launch template: template vừa tạo.
    • VPC/subnets: public app subnets cho demo low-cost.
    • Attach to existing target group.
    • Desired: 1, Min: 1, Max: 4.
    • Scaling policy: Target tracking CPU 70%.
  5. Đợi target group hiện trạng thái Healthy.

Kết quả cần có

ALB nhận request, target group healthy, ASG tự quản lý EC2 chạy API.

Ảnh cần chụp

  • ALB DNS và listener.
  • Target group health check path /health.
  • Target healthy.
  • Launch template IAM role và security group.
  • ASG desired/min/max và scaling policy.

Kiểm tra nhanh

Mở http://ALB-DNS/health. Nếu không trả 200, kiểm tra service API, port 8080 và SG.

Bước 10

Cấu hình CloudWatch logs, metrics, dashboard và alarms

Monitoring
  1. Vào CloudWatch → Log groups, kiểm tra log group của API nếu EC2 đã gửi logs.
  2. Vào Dashboards → Create dashboard tên omnistay-dashboard.
  3. Thêm widgets:
    • ALB RequestCount, TargetResponseTime, HTTPCode_Target_5XX_Count.
    • EC2 CPUUtilization và ASG GroupDesiredCapacity/GroupInServiceInstances.
    • RDS CPUUtilization, DatabaseConnections, FreeStorageSpace.
    • ElastiCache CPUUtilization, CurrConnections, CacheHits, CacheMisses nếu có metric.
  4. Tạo alarm demo:
    • ASG scale-out dựa trên CPU 70%.
    • ALB 5xx tăng bất thường.
    • RDS CPU cao hoặc storage thấp nếu muốn.

Kết quả cần có

Dashboard chứng minh hệ thống quan sát được và Auto Scaling có metric.

Ảnh cần chụp

  • CloudWatch dashboard đầy đủ widgets.
  • Log group có log API.
  • Alarm hoặc scaling policy liên quan CPU.

Bước tùy chọn

NAT Gateway, VPC Endpoints, Route 53, ACM và SES

Production-like
  1. NAT Gateway: chỉ tạo nếu EC2 app nằm private subnet và cần outbound internet để tải package/artifact. Lưu ý NAT Gateway tốn phí theo giờ.
  2. VPC Endpoints: dùng thay NAT cho S3, SSM, CloudWatch Logs nếu muốn private traffic. S3 Gateway endpoint thường tiết kiệm hơn interface endpoint.
  3. Route 53: dùng nếu có domain riêng. Có thể bỏ qua và dùng CloudFront domain mặc định.
  4. AWS Certificate Manager: tạo certificate public nếu dùng domain HTTPS riêng cho CloudFront hoặc ALB.
  5. Amazon SES: tùy chọn gửi email xác nhận booking. Không cần cho demo hạ tầng tối thiểu.

Chi phí

NAT Gateway, Route 53 hosted zone và một số endpoints có phí riêng. Chỉ bật khi cần demo production-like.

Ảnh cần chụp nếu dùng

  • NAT Gateway available.
  • VPC endpoints route/DNS.
  • Route 53 record trỏ CloudFront.
  • ACM certificate issued.
  • SES verified identity.

4. Cleanup sau buổi lab

Không bỏ qua bước này. Sau khi demo xong, kiểm tra Billing và xóa hoặc stop tài nguyên tốn phí.
  • Delete CloudFront distribution nếu không dùng nữa. Có thể mất thời gian disable trước khi delete.
  • Delete ALB và target group.
  • Scale ASG về 0 hoặc delete ASG.
  • Terminate EC2 không còn dùng.
  • Stop RDS nếu còn cần dữ liệu, hoặc snapshot rồi delete nếu kết thúc lab.
  • Delete ElastiCache Redis/Valkey cluster.
  • Delete NAT Gateway nếu đã tạo.
  • Delete unattached Elastic IP nếu có.
  • Xóa S3 objects trước khi delete bucket.
  • Kiểm tra CloudWatch log retention để tránh giữ log quá lâu.
  • Giữ AWS Budget để còn cảnh báo chi phí.
